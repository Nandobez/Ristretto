ristretto — java fullstack CLI · MIT license · alias: rist / r

One shot.
Three espressos.

rist bundles jdp (deps) + Xpresso (backend) + Macchiato (frontend) into a single CLI, and adds a fullstack lifecycle on top: scaffold, run, inspect, and hit your API — all through one command. 

$ curl -fsSL https://raw.githubusercontent.com/Nandobez/Ristretto/main/install.sh | bash
download one tool, drive all three rist <command>
jdp dependency management add · search · why · doctor --fix (CVE via OSV.dev) · unused · weight · migrate maven→gradle xpresso backend — Spring Boot rails-style generators: resource · model · seed · auth · migration — plus db, routes, beans, health macchiato frontend — React @Page/@Model codegen → .tsx + types + routes · vite hot reload · served with the backend

verbs that exist in more than one tool? target it explicitly: rist macc add Button

rist — demo

60-second backend

six commands, zero config 
$ rist new shop --group io.acme --no-frontend
$ cd shop
$ rist resource Order item:string qty:int paid:bool
  # full CRUD, tests, error handler
$ rist g seed Order item:string qty:int paid:bool
  # fake data on dev startup
$ rist up      # build + run detached
$ rist api GET orders   # hit the live API
$ rist down

rist resource generates a complete, compiling CRUD slice:

  • domain/Order.java — JPA entity
  • repository/OrderRepository.java
  • dto/OrderRequest.java validated, no id + OrderResponse.java
  • service/OrderService.java — @Transactional
  • web/OrderController.java — DTO-based, paged
  • exception/GlobalExceptionHandler.java — 400 / 404 / 500
  • db/migration/V…__create_orders.sql — Flyway
  • …ServiceTest + …ControllerTest

state lives under .ristretto/ — git-ignored automatically

Run & managerist up · status · logs · down

lifecycle for the app rist started — state under .ristretto/

rist up [--profile dev] [--port N] [--build] [--solve]package (if needed) + run detached; ready banner with Local / Network / Swagger / Health URLs. --solve auto-fixes common blockers (duplicate migrations, ordering FKs, missing ;/imports, boot-time NPE) then retries
rist reload [--solve]stop + fresh build + start — pick up code changes in one command (alias restart)
rist statusrunning (pid · uptime) with the same banner, or stopped
rist logs [-f] [-n N] [--raw]app logs reformatted time · level · message
rist downstop the app started by up
rist servedev mode: backend and frontend in parallel

Talk to your APIrist api

api reads the running app's /v3/api-docs, so -m bodies always satisfy the real request schema

$ rist api GET orders                     # aligned table + pagination
$ rist api GET orders/1                   # single record
$ rist api POST -m orders                 # -m fills a valid mock body from OpenAPI
$ rist api POST -m orders item=Cafe qty=3 # key=value overrides
$ rist api PUT -m orders/1
$ rist api DELETE orders/1
$ rist api GET orders --raw               # raw (colored) JSON

Consume any API--client · --discover · --try

point rist at an external API and it generates a typed backend client — a record model, a Spring RestClient, and a controller that re-exposes it under /api/…. Keys and JWT logins are detected and kept in .env, never in code

$ rist api --client Product --url https://fakestoreapi.com/products  # GET → model + client + controller
$ rist api --client Post --curl                    # paste a curl; method/auth/body detected
$ rist api --discover --url https://petstore3.swagger.io/api/v3     # auto-detect every endpoint + fields
$ rist api --client Pet --discover --url … --pick "POST /pet"       # generate one from the schema
$ rist api --client Me --url …/auth/me \
    --login …/auth/login --login-body '{"username":"$USER","password":"$PASS"}'  # login → JWT → Bearer, re-login on 401
$ rist api --try                                   # interactive Postman: fuzzy pick, fill body, formatted/raw
